## 4) 数据处理方法

### 4.1 classic（标准 CVRP）处理方法

- 输入：CVRPLib `A/B/E/F/M/P` 的原始 `.vrp/.sol`
- 解析字段：节点、需求、仓库、容量、距离定义、参考解
- 兼容两类距离格式：
  - `EUC_2D`
  - `EXPLICIT + LOWER_ROW`
- 输出：统一 JSON 格式（`base/meta`）+ 索引表 + QA 报告

### 4.2 fresh（生鲜扩展）处理方法

基于 classic 的 `base.json` 扩展业务字段（不破坏原始 CVRP 基础结构）：

- 服务时长：`service_time_min = 5 + ceil(0.2 * demand)`
- 时间窗：按“到仓库距离分层 + 抖动”生成
- 易腐等级：按需求分位数（30/70）映射 low/medium/high
- 最大在途时长：low=300, medium=180, high=90（分钟）
- 温区：按易腐等级概率采样（frozen/chilled/ambient）
- 惩罚项：
  - `late_penalty_per_min = 0.8 * demand`
  - `spoilage_penalty = 2.0 * demand * freshness_weight`
- 可复现：全局种子 `5491` + 实例级稳定子种子

## 6) fresh 数据集的算法使用说明（详细版）

下面给出 `fresh` 新增字段在建模和算法中的具体用法。算法同学可以从“先简单可跑”到“逐步加约束”分阶段接入。

### 6.1 建议的求解阶段

- 阶段 1（最小可跑）：只用 `distance_matrix + vehicle_capacity + demand`，先复用 classic 求解器。
- 阶段 2（时效约束）：加入 `ready_time_min/due_time_min + service_time_min`，形成带时间窗版本。
- 阶段 3（生鲜品质）：再加入 `freshness_class + max_travel_time_min`，限制在途时长。
- 阶段 4（业务代价）：加入 `late_penalty_per_min + spoilage_penalty + objective_weights`，做多目标或加权单目标优化。

### 6.2 新增字段逐项用途

- `distance_to_depot`
  - 含义：客户到仓库的基础距离。
  - 用途：可作为聚类或初始化分区特征（例如先服务近距离高时效客户）。

- `distance_band` (`near/mid/far`)
  - 含义：客户距离分层标签。
  - 用途：构造分层调度策略（near 可后置，far 可前置）或候选点优先级规则。

- `service_time_min`
  - 含义：车辆在客户点卸货/签收耗时。
  - 用途：更新到达时间递推，计算是否迟到；在邻域搜索中用于快速可行性检查。

- `ready_time_min`, `due_time_min`
  - 含义：客户可服务时间窗。
  - 用途：构成时间窗可行性约束；用于定义迟到量 `lateness_i = max(0, arrival_i - due_i)`。

- `freshness_class` (`high/medium/low`)
  - 含义：订单易腐等级。
  - 用途：用于差异化策略（例如 high 优先插入更早路线，或使用更严格局部搜索接受准则）。

- `max_travel_time_min`
  - 含义：该客户可接受的最大在途时长。
  - 用途：定义保鲜超时量 `spoil_i = max(0, travel_time_i - max_travel_i)`，作为硬约束或惩罚项。

- `temp_zone` (`frozen/chilled/ambient`)
  - 含义：温控需求。
  - 用途：若后续扩展异构车队，可作为车辆-订单匹配约束；当前也可用于分组统计不同温区服务质量。

- `late_penalty_per_min`
  - 含义：每分钟迟到成本系数。
  - 用途：迟到惩罚项 `P_late = Σ lateness_i * late_penalty_per_min_i`。

- `spoilage_penalty`
  - 含义：超保鲜时长对应的单位惩罚强度。
  - 用途：保鲜惩罚项 `P_spoil = Σ spoil_i * spoilage_penalty_i`。

- `problem_variant`
  - 含义：问题标记（`CVRP_FRESH`）。
  - 用途：统一数据加载入口时自动切换求解模式。

- `depot_start_time_min`
  - 含义：仓库发车基准时间（分钟制）。
  - 用途：用于路线时间推演起点，和 `service_time_min/ready/due` 一起计算到达时刻。

- `objective_weights`
  - 含义：距离、迟到、保鲜三类目标权重。
  - 用途：统一总目标函数，便于调参比较不同策略。

### 6.3 建议目标函数（可直接实现）

设：

- `D`：总行驶距离
- `P_late = Σ_i max(0, arrival_i - due_i) * late_penalty_per_min_i`
- `P_spoil = Σ_i max(0, travel_time_i - max_travel_time_i) * spoilage_penalty_i`

则可用加权目标：

- `Obj = w_d * D + w_l * P_late + w_s * P_spoil`

其中 `w_d, w_l, w_s` 可先读 `objective_weights`，后续在验证集上调参。

### 6.4 约束接入建议

- 容量约束：保持 classic 约束不变。
- 时间窗约束：
  - 硬约束方案：`arrival_i <= due_i` 必须满足，不满足直接判 infeasible。
  - 软约束方案：允许迟到，但计入 `P_late`。

- 保鲜约束：
  - 硬约束方案：`travel_time_i <= max_travel_time_i`。
  - 软约束方案：超时部分计入 `P_spoil`。

建议先“软约束”跑通，再逐步切到“硬约束”做对比实验。

### 6.5 评估指标建议（fresh）

- 路径相关：总距离 `D`、平均每车里程。
- 时效相关：迟到客户数、总迟到分钟、平均迟到分钟。
- 品质相关：超保鲜客户数、总超时分钟。
- 业务相关：`P_late`、`P_spoil`、总目标 `Obj`。
- 可行性：容量违约率、时间窗违约率、保鲜违约率。
